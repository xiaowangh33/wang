# Wheeled Dog Realtime Control Notes

## Frequency Separation

The deployment stack should keep three timing layers independent:

1. Policy inference updates high-level setpoints.
2. The hardware or simulation control layer recomputes actuator commands from the latest state.
3. The bus or simulator step sends/applies the most recent low-level command.

For the current wheel-legged dog MuJoCo validation setup:

- MuJoCo timestep: `0.005 s` / `200 Hz`
- Policy decimation: `4`
- Policy inference rate: `50 Hz`
- Leg command form: `tau = kp * (q_des - q) + kd * (dq_des - dq) + tau_ff`
- Wheel command form in MuJoCo: velocity actuator target `dq_des`

The policy should update `[kp, q_des, kd, dq_des, tau_ff]` at the training-compatible rate, then the low-level layer should hold those setpoints and recompute/apply control at its own rate.

## Hardware Porting Rule

When moving this framework to the real robot, do not make the policy frequency depend directly on the hardware PD frequency.

Current hardware structure:

- Policy thread: fixed by elapsed time, `20 ms` / `50 Hz`
- Hardware state-machine tick: `5 ms` / `200 Hz`
- PC-to-MCU setpoint bridge: `5 ms` / `200 Hz`, carrying all 16 five-tuples
- MCU control task: `1 ms` / `1 kHz`, holding the latest PC setpoint
- Four independent 1 Mbps CAN buses: each bus queues two of its four motors per
  1 ms tick, so each motor gets a command every `2 ms` / `500 Hz`

The 200 Hz transport does not run ONNX at 200 Hz. A policy result is generated
once per 20 ms training-aligned interval and the same five-tuple is republished
on the four intervening 5 ms transport ticks. `WHEELDOG_MCU_SETPOINT_HZ` remains
available for diagnostics, but its deployment default is 200 Hz and it never
changes `kPolicyPeriodSec`.

Do not infer motor refresh from the MCU task counter. The firmware reports
per-motor CAN TX queued/completed/deferred counters; the PC bridge computes the
actual motor command rate from TX-complete interrupts and fails safe if it stays
below the configured threshold.

The firmware now treats command TX and fresh feedback as two independent
requirements. During live startup it sends only zero commands, counts each
motor's `COMM_OPERATION_STATUS (0x02)` replies, and cross-checks their decoded
position/velocity against `READ_PARAM mech_pos/mech_vel`. MCU PD is armed only
after all local motors pass the comparison and maintain at least 450 Hz; the PC
bridge independently differences both TX-complete and operation-status counters
for all 16 motors. PD consumes each feedback generation once, so a 1 kHz task
never recomputes from the same held sample.

The comparison tolerances are qualification settings, not assumed protocol
facts: `0.05 rad` position, `0.20 rad/s` velocity, three consecutive matches,
and a `5 ms` freshness deadline. Hardware startup is intentionally static: the
bridge sends zero commands and requires 16/16 online, absolute-limit mode,
fresh >=450 Hz operation status and matching READ_PARAM position/velocity.
No excitation command is generated by the deployment bridge. The existing
position/velocity excitation masks remain diagnostic only and are not readiness
gates. Consequently a static startup proves delivery and current-pose channel
agreement, but does not re-prove dynamic velocity scale or direction; those
were established on the powered bench and are checked mechanically during the
suspended `z` direction test.

The standalone dry-run tool still has an explicit one-motor qualification flag
for diagnostics. It is never set by the deployment bridge.

## Coherent 200 Hz Transport / 50 Hz Policy Observation

The two MCU USB feedback streams must not be combined merely by arrival order.
For every PC setpoint sequence, each MCU now latches one observation after
draining CAN RX. Each local motor position is extrapolated from its latest
qualified `q/dq` to that MCU snapshot tick, and its original sample age is
reported. The PC bridge keeps a bounded 64-epoch history per MCU and joins
feedback by `observation_seq`; it does not require both ports' latest packets
to be the same epoch. It publishes to shared memory only when:

- MCU A and MCU B report the same nonzero observation sequence;
- all local fast-feedback validity bits remain set (the MCU's effective stale
  boundary remains `5 ms`) and reported sample age is at most `10 ms`;
- the two matching USB feedback packets arrive within `30 ms`;
- the matched packets have spent at most `100 ms` in PC-side history.

If no new qualifying common epoch is produced for `500 ms`, the bridge and C++
runtime latch stop control. This remains independent from the MCU's `300 ms`
command watchdog and does not allow stale samples to re-enter local PD.

Shared memory reports `observation_seq`, `observation_max_sample_age_ms`,
`observation_bridge_skew_us`, per-motor sample ages and dropped observation
epochs. The policy consumes the latest accepted coherent snapshot only on its
50 Hz elapsed-time schedule. This prevents a policy observation from containing
eight joints from epoch N and eight joints from epoch N+1. Exact physical
simultaneity still
requires a shared hardware trigger or synchronized MCU clock; the current
software guarantee is a measured, bounded-skew coherent epoch.

## Important Policy Timing Caution

The deployed RL state uses an elapsed-time schedule rather than deriving policy
frequency from the hardware tick:

- target policy period: `0.02 s`
- run ONNX only when `now - last_policy_time >= 0.02 s`
- keep the last policy setpoint between policy updates

This preserves the training-compatible `50 Hz` policy while the hardware tick
and PC-to-MCU transport run at `200 Hz`, the MCU control task runs at `1 kHz`,
and the per-motor CAN command refresh runs at `500 Hz`.

## Wheel Actuator Handling

The framework distinguishes actuator modes in `robot_model_config.hpp`:

- `PositionPD`: leg joints, position target converted to torque by low-level PD
- `Velocity`: wheel joints, velocity target

For MuJoCo, wheel joints are modeled as `<velocity>` actuators, so the deployment layer sends `dq_des` directly to the simulator actuator.

For hardware, choose one of these implementations:

- If the wheel drive supports velocity mode, send `dq_des` to the drive velocity controller.
- If the wheel drive is torque-only, implement a wheel speed loop in the hardware layer:

```cpp
tau_wheel = kd_wheel * (dq_des - dq);
```

Keep the policy output semantics unchanged: the policy still produces wheel velocity targets.

The deployed RS01 path uses communication type 1 motion-control mode rather
than the drive's separate speed-mode PI loop. Wheel phase is continuous in
software and is unwrapped with the confirmed single-turn encoder period of
`2*pi`; no position wall is applied to a wheel.

For a wheel, policy `dq_des` is a virtual target in
`tau = 0.4 * (dq_des - dq)`, not permission for the wheel to reach that speed.
Hardware caps this virtual target at `+/-20 rad/s` and retains the rated
`17 Nm` peak limit. The native RS01 CAN velocity field stays encoded over its
documented `+/-44 rad/s` range; that codec range is not a deployment command
limit. The MCU converts the virtual target to a torque-limited motor target on
every fresh 500 Hz feedback sample. The former wheel-only `600 Nm/s` torque
ramp and `8 -> 10 rad/s` measured-speed derate/trip are disabled; the drive's
native protections remain active.

Leg desired speed remains capped at `8 rad/s`. Stand-up commands all wheels to
`0 rad/s` with zero travel. The editable `kStandWheelTravelRad` and
`kStandWheelVelocityLimitRadps` constants in
`description/robot_model_config.hpp` control optional future stand-up wheel
motion. MCU calibration handles motor mounting polarity so all four
joint-space wheel velocities retain the same URDF convention. During the
current deployment, HipX, HipY, and Knee all return to the training-aligned
`Kd = 2` through the existing two-second RL-entry interpolation. The defaults
are in `description/robot_model_config.hpp`; `RL_HIP_KD` and `RL_KNEE_KD`
remain available as hardware-only runtime overrides for supported A/B tests.

The RS06 leg effort ceiling is `36 Nm` on both PC and MCU, matching the training
actuator effort limit. Hardware RL also prints `[TorqueAlign]` at 5 Hz. It
evaluates the command held
during the preceding policy interval at the current measured state and compares
that estimate with current motor torque feedback. The estimate includes the
PC/MCU torque ceiling, but not a raw-position-wall intervention.

The extended 1172-byte MCU feedback includes `final_joint_torque_cmd_nm[]` and
the low-rate per-motor VBUS telemetry: the torque array is the
logical-joint command after torque ceiling, raw-position wall and leg-only
measured-speed guard. `[WDP4 torque actual]` compares that value with motor torque feedback at
an explicitly configured diagnostic rate. A live safety stop latches the
pre-zero command for the fatal report. Normal deployment instead prints all 16
motor temperatures and bus voltages every three seconds.
